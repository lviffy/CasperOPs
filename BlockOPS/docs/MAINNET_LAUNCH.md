# Casper Mainnet Launch Checklist (v1.0.0)

> **Audience:** the team lead driving the v1.0 mainnet launch.
> Print this, work through every box, sign + date each one.

This checklist assumes you've already completed Phase 28
([Live Testnet Deployment](../PHASE28_RUNBOOK.md)) and tagged the
release as `v1.0.0-rc.1`.

## Pre-launch (T-7 days)

### Code & config

- [ ] All Phase 26-29 tests green on CI (227+ backend, 39 frontend,
      64 cargo, 25+ MCP, 22 Playwright E2E)
- [ ] No `TODO` / `FIXME` markers in the launch-blocking surfaces
      (`backend/app.js`, `backend/services/`, `frontend/app/`,
      `n8n_agent_backend/dispatcher.py`)
- [ ] `.env.example` has the v1.0.0-rc.1 contract hashes as defaults
- [ ] `frontend/lib/contracts.ts` matches the v1.0.0-rc.1 contract hashes
- [ ] `n8n_agent_backend/tools/schema.json` matches
- [ ] `CASPER_NETWORK` env var is set to `mainnet` in the production
      secret store (Fly.io / Render / Vercel)
- [ ] `CASPER_MAINNET_RPC_URL` set to `https://rpc.mainnet.casperlabs.io/rpc`
- [ ] `CSPR_CLOUD_API_KEY` set to a paid-tier key (free tier rate-limits
      will 429 us at scale)
- [ ] `CASPER_MAINNET_*_HASH` populated for all 6 contracts

### Security

- [ ] `ADMIN_SECRET` is 32+ chars (auto-generated by the host)
- [ ] `MASTER_API_KEY` is 32+ chars (auto-generated by the host)
- [ ] `METRICS_TOKEN` is 32+ chars (auto-generated by the host)
- [ ] `METRICS_ALLOWED_CIDRS` excludes `0.0.0.0/0`
- [ ] All secrets verified to NOT be the testnet defaults
- [ ] `JWT_SECRET` (if used) is 32+ chars and rotated from the testnet one
- [ ] `docs/security-audit.md` reviewed + signed off by the security lead

### Infrastructure

- [ ] DNS: `blockops.example`, `api.blockops.example`, `mcp.blockops.example`
      all point at the production load balancer (Cloudflare in front)
- [ ] SSL certs: all 3 hosts have valid Let's Encrypt certs (Cloudflare
      auto-renews, but verify in the dashboard)
- [ ] CDN: Cloudflare in front of the backend (caching `/v1/tools`
      catalog + the static frontend assets)
- [ ] Sentry DSNs populated for backend (`SENTRY_DSN`) + frontend
      (`NEXT_PUBLIC_SENTRY_DSN`)
- [ ] Sentry alert rules configured per [`OPERATIONS.md` §2](./OPERATIONS.md#2-sentry-alert-rules)
- [ ] Uptime monitor configured per [`OPERATIONS.md` §3](./OPERATIONS.md#3-uptime-monitoring)
- [ ] Log shipping configured per [`OPERATIONS.md` §4](./OPERATIONS.md#4-structured-log-shipping)

## Launch day (T-0)

### Final deploy

- [ ] `./scripts/deploy-mainnet.sh --dryrun` → review the plan
- [ ] `./scripts/deploy-mainnet.sh --confirm` → type `deploy-mainnet`,
      capture the 6 contract hashes
- [ ] Update `backend/.env` with the new `CASPER_MAINNET_*_HASH`
      values
- [ ] `./scripts/deploy-backend.sh prod` → fly deploy the backend with
      `CASPER_NETWORK=mainnet`
- [ ] `./scripts/deploy-frontend.sh prod` → vercel deploy the frontend
- [ ] `./scripts/deploy-mcp.sh deploy` → render deploy the MCP server
- [ ] Verify all 3 services respond on their mainnet hosts
      (`curl https://api.blockops.example/health/ready`)

### Smoke tests

- [ ] `curl https://api.blockops.example/v1/tools` → returns 22 tools
- [ ] Sign a paid tool deploy via CSPR.click + verify x402 flow
- [ ] Watch the Sentry dashboard for 5 minutes — should be quiet
- [ ] Watch the Grafana dashboard for 5 minutes — `blockops_http_requests_total`
      should be ticking, no spikes

### Communications

- [ ] Twitter / X launch post drafted (include screenshot of the
      v1.0 dashboard)
- [ ] Discord launch post drafted
- [ ] GitHub release `v1.0.0` published with:
  - [ ] Binary checksums (the on-chain deploy hashes + their
        corresponding WASM file SHA-256)
  - [ ] Changelog (one section per phase 26-29)
  - [ ] Migration guide for anyone on v0.x
- [ ] Email to the existing waitlist (template in
      `docs/LAUNCH_CHECKLIST.md` §6)

## Post-launch (T+1 to T+7)

### Operational

- [ ] Status page shows 🟢 Operational for every component
- [ ] First 24h: monitor for Sentry spikes every 2 hours
- [ ] First 7 days: review every Sentry issue, classify (bug vs.
      expected noise), file follow-ups
- [ ] Backup the mainnet deployer key in cold storage (1Password
      + offline USB)
- [ ] Decommission the testnet deployer key (move to cold storage,
      do NOT delete — the v1.0.0-rc.1 events on CSPR.cloud are
      linked to it)
- [ ] Rotate the production `ADMIN_SECRET` 7 days after launch
      (gives the deploy scripts time to settle)

### Marketing

- [ ] HackerNews post (timed for US-morning peak)
- [ ] ProductHunt launch
- [ ] Two follow-up posts on the team blog:
  - [ ] "Why Casper" — the design rationale for the chain choice
  - [ ] "x402 on Casper" — how the payment protocol landed

### Internal

- [ ] Retro on the v1.0 launch (2-hour meeting, all team members)
- [ ] Document lessons learned in `docs/retros/2026-Q3-v1-launch.md`
- [ ] Vote on the next quarter's priorities (per retro outcomes)

## Rollback

If something critical breaks within the first hour:

1. Freeze the public docs site (CloudFlare "Under Attack Mode" toggle)
2. Update `docs/STATUS.md` + the hosted status page with the incident
3. Run the rollback per [`RUNBOOK.md` §9](./RUNBOOK.md#9-rollback-a-deploy)
4. Post a public incident within 30 minutes
5. Schedule a post-mortem within 24 hours

If something critical breaks within the first week:

1. Revert to `v1.0.0-rc.1` (last known-good testnet-tagged build)
2. Re-deploy to mainnet
3. Investigate offline; do NOT push the fix until the post-mortem is signed off

## Success metrics (T+30)

If we're on track:

- 1000+ unique API keys issued
- 50+ paying customers
- 99.5%+ backend availability
- < 1 Sentry P0/P1 incident unresolved after 24h
- 10+ community-built agents on the marketplace

If we're not:

- File an honest retro: what did we miss, what would we do differently?
- Reset the metrics and re-measure after the next quarter.
