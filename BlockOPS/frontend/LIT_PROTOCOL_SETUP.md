# Lit Protocol Setup

BlockOps uses Lit Protocol V1 (Naga) for decentralized key management. The app supports both PKP-based wallets and Lit-encrypted traditional wallets, so users can create a seedless agent wallet or continue using an imported EOA without storing raw private keys in plaintext.

## Feature Context

BlockOps is built around decentralized key management for autonomous AI agents. The core user experience shift is from "paste your private key into the app" to "create a seedless programmable wallet where key material is never exposed."

This setup supports a hybrid wallet model that keeps compatibility with imported EOAs while making PKP the default and recommended path.

## What We Implemented

- PKP wallet minting on Lit Naga testnet.
- PKP-backed transaction signing for prepared blockchain actions.
- Lit-encrypted storage for traditional private keys using the `lit:v1:` payload format.
- Shared Lit wallet support across the web app and backend-driven execution paths.
- Runtime routing by `wallet_type` so PKP and traditional flows can coexist safely.
- Custom Lit Actions (`ENCRYPT_ACTION_CODE` and `DECRYPT_ACTION_CODE`) for compatibility-mode key protection.
- Cross-surface wallet reuse between web UI flows and backend/Telegram execution.

## Wallet Modes in BlockOps

### 1. PKP Wallet Mode

This is the main Lit Protocol flow for the project.

- A PKP is minted on Lit Naga testnet.
- The app stores only public wallet metadata.
- Transactions can later be signed through the PKP path without exposing private key material.

Stored fields for PKP wallets:

- `wallet_type = 'pkp'`
- `wallet_address`
- `pkp_public_key`
- `pkp_token_id`

What is not stored for PKP wallets:

- no PKP private key
- no seed phrase
- no raw key material generated by the app

### 2. Traditional Wallet Mode

This mode is available for compatibility with imported wallets and earlier flows.

- The user imports or generates a standard EOA private key.
- The key is encrypted before storage.
- The encrypted payload is stored in `users.private_key`.
- Decryption happens only when a signing flow explicitly needs it.

This compatibility path uses custom Lit Actions:

- `ENCRYPT_ACTION_CODE`
- `DECRYPT_ACTION_CODE`

## Technical Architecture Notes

### High-level architecture

1. Frontend (Next.js): onboarding, wallet setup, chat and contract interaction UI.
2. Backend (Node.js): Telegram integration, transaction orchestration, AI routing, server-side PKP signing.
3. Persistence (Supabase): user profile records, wallet metadata, and agent-linked state.

### Network architecture

- Lit Naga testnet: PKP minting, Lit auth, and PKP signing infrastructure.
- Arbitrum Sepolia: blockchain transaction broadcast in current app flows.

This separation keeps key management decentralized in Lit while preserving current chain execution behavior.


## Required Environment Variables

Add these to the frontend server environment, for example `frontend/.env.local`:

```bash
LIT_API_BASE_URL=https://api.dev.litprotocol.com/core/v1
LIT_USAGE_API_KEY=your_lit_usage_api_key
LIT_PKP_ID=your_default_lit_pkp_id
LIT_PKP_CONTROLLER_PRIVATE_KEY=your_controller_private_key
LIT_NAGA_HANDSHAKE_TIMEOUT_MS=45000
LIT_AUTH_STORAGE_PATH=optional_local_auth_cache_path
LIT_APP_NAME=blockops
```

Backend-driven signing paths should also have matching Lit configuration in `backend/.env`:

```bash
LIT_API_BASE_URL=https://api.dev.litprotocol.com/core/v1
LIT_USAGE_API_KEY=your_lit_usage_api_key
LIT_PKP_ID=your_default_lit_pkp_id
LIT_PKP_CONTROLLER_PRIVATE_KEY=your_controller_private_key
LIT_NAGA_HANDSHAKE_TIMEOUT_MS=45000
```

Notes:

- `LIT_USAGE_API_KEY` must be allowed to execute the Lit Actions used by this project.
- `LIT_PKP_CONTROLLER_PRIVATE_KEY` is used by the server-side PKP flow.
- `LIT_PKP_ID` is a fallback PKP identifier for encrypted-key flows when the payload does not carry its own `pkpId`.


## Storage Format in `users.private_key`

Traditional wallets are stored as Lit-encrypted ciphertext.

Format:

- Prefix: `lit:v1:`
- Payload: JSON containing `version`, `provider`, `pkpId`, `ciphertext`, and `createdAt`

Example:

```text
lit:v1:{"version":1,"provider":"lit-chipotle","pkpId":"0x...","ciphertext":"...","createdAt":"..."}
```

## Runtime Signing Flow

At execution time, BlockOps branches on the wallet type:

- `wallet_type = 'pkp'`
  - use the PKP signing path
  - create a Lit-authenticated PKP signer
  - sign and broadcast the transaction
- `wallet_type = 'traditional'`
  - decrypt the `lit:v1:` payload
  - reconstruct the EOA signer
  - sign through the legacy flow

This allows the same product to support both decentralized PKP wallets and encrypted imported wallets.

## Cross-Platform Wallet Usage

The same Lit-backed wallet metadata is intentionally reused across product surfaces:

- Web frontend: wallet setup modal, contract interaction UI, and agent chat execution.
- Backend and Telegram flows: server-side transaction execution and direct tool routing.

This proves the wallet is part of real autonomous execution, not only a UI-only feature.

## How to Test the Lit Flow

1. Start the frontend and backend services.
2. Sign in to BlockOps.
3. Open the wallet setup modal.
4. Confirm the highlighted default option is `Create PKP Wallet`.
5. Use the `Create PKP Wallet` flow and click `Generate PKP`.
6. Confirm the app stores `wallet_type`, `wallet_address`, `pkp_public_key`, and `pkp_token_id`.
7. Confirm no raw private key is stored.
8. Trigger a blockchain action from the chat UI or contract interaction UI.
9. Confirm signing uses the PKP path.

To test the compatibility flow:

1. Switch to the traditional wallet path.
2. Import or generate a standard EOA.
3. Confirm the database stores a `lit:v1:` payload instead of plaintext.
4. Trigger a signing action and verify decryption happens only at execution time.

To test cross-platform PKP reuse:

1. Link the same user to the Telegram/backend flow.
2. Trigger an action that requires signing.
3. Verify the backend detects `wallet_type = 'pkp'`.
4. Confirm backend execution uses PKP signing instead of private-key decryption.

## Why Lit Matters Here

Lit gives BlockOps a better wallet model for AI agents:

- users can create a seedless agent wallet
- imported keys are never stored in plaintext
- signing can happen in both web and backend contexts
- wallet management fits autonomous execution better than browser-only signing
