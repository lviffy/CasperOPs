# x402 Payment Protocol — Casper Implementation

BlockOps uses the **x402 pattern** ("HTTP 402 Payment Required") to monetize agent
tool invocations. The client pays per call in **CSPR via a Casper deploy**; the
backend verifies the deploy on-chain before executing the tool.

This spec is intentionally minimal — a single deploy-template per tool, a single
verification header, and a single retry flow. The full payment protocol
documentation lives alongside this spec.

## 1. HTTP 402 challenge

When a paid tool is invoked without a valid `X-Casper-Payment-Deploy-Hash`
header, the backend responds:

```http
HTTP/1.1 402 Payment Required
Content-Type: application/json

{
  "toolId": "register_agent",
  "priceCspr": "0.50",
  "priceMotes": "500000000",
  "payToPublicKey": "010101010101010101010101010101010101010101010101010101010101010101",
  "deployTemplate": {
    "contractHash": "hash-<64hex>",
    "entryPoint": "transfer",
    "args": {
      "recipient": "010101010101010101010101010101010101010101010101010101010101010101",
      "amount": "500000000"
    }
  },
  "memo": "BlockOps tool payment: register_agent"
}
```

The `deployTemplate` is a fully-formed Casper deploy JSON the client signs with
its CSPR.click wallet and broadcasts to the testnet. The CEP-18 `transfer`
entry point is the canonical payment surface (CSPR or any CEP-18 token can be
used; the same `entryPoint = "transfer"` works for either).

## 2. Client payment

The client signs the deploy template via CSPR.click:

```ts
const signed = await signDeploy(challenge.deployTemplate, activePublicKey)
const result = await sendDeploy(challenge.deployTemplate, activePublicKey)
const deployHash = result.deployHash
```

The client retries the original tool request with:

```http
POST /v1/tools/register_agent
X-Casper-Payment-Deploy-Hash: <deployHash>
X-Casper-Payment-Payer-PublicKey: <activePublicKey>
Content-Type: application/json
```

## 3. Backend verification

The backend middleware (`backend/middleware/x402-verify.js`):

1. Parses `X-Casper-Payment-Deploy-Hash` and `X-Casper-Payment-Payer-PublicKey`.
2. Looks up the deploy via `info_get_deploy` on the Casper RPC.
3. Confirms:
   - The deploy is **executed** (in a block, no `error_message`).
   - The payment recipient matches `payToPublicKey`.
   - The payment amount is **≥** `priceMotes` for `toolId`.
   - The payer public key matches `X-Casper-Payment-Payer-PublicKey`.
4. Returns 200 if all checks pass; 402 with a fresh challenge otherwise.

The middleware caches verified deploys in-memory for 5 minutes so the same
deploy hash is not re-verified on every retry.

## 4. Free tools

Read-only tools (`get_balance`, `get_token_info`, `get_nft_info`,
`lookup_deploy`, `lookup_block`, `fetch_price`, `calculate`, `get_reputation`)
are listed in the free tier in [`TOOL_PRICING`](#5-tool-pricing-table) and
**never** trigger a 402. The middleware short-circuits for them.

## 6. Optional: JWT execution token

After a successful payment, the backend can optionally issue a short-lived JWT
execution token so the client can call multiple paid tools without paying
again. The token:

- Encodes `payerPublicKey`, `toolIds[]`, `expiresAt`.
- Is signed with `JWT_SECRET` (HS256, 1-hour TTL).
- Must be passed as `X-Casper-Execution-Token: <jwt>` on subsequent calls.

`scripts/issue-execution-token.js` is included to mint these tokens manually
for testing.

## 7. Frontend helper

[`frontend/lib/x402-client.ts`](./../frontend/lib/x402-client.ts) wraps any
`fetch` call so paid tool invocations automatically:

1. Detect a 402 response.
2. Open the CSPR.click popup to sign the deploy template.
3. Broadcast via `sendDeploy`.
4. Retry the original request with the `X-Casper-Payment-Deploy-Hash` header.

The helper also surfaces the deploy hash + explorer link in a toast
(`DeployStatusIndicator`) so users can verify the payment on-chain.

## 8. Example flow

```ts
// 1. Client calls a paid tool.
const response = await x402Fetch("https://api.blockops.dev/v1/tools/register_agent", {
  method: "POST",
  body: JSON.stringify({ agentId: "agent-1", metadataUri: "ipfs://..." }),
})

// 2. If 402, x402-client auto-handles signing + retry.
// 3. On success, response.json() contains the normal tool result.
```
